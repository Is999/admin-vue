# 需求歧义信号

## 读源码后可直接推进

请求范围很窄、目标文件或函数明显、预期行为能直接从代码或文档验证时，可以不追问，读完源码后推进。

示例：

- 用户给出准确文件路径、函数、route、SQL literal 或失败命令。
- 用户要求执行已经定义好的机械传播。
- 仓库文档和附近测试已经清楚定义预期行为。

## 需要暂停确认

猜测会导致错误行为或浪费实现时，编辑前暂停确认：

- `优化一下`, `处理一下`, `支持一下`, `看下这个问题`, `感觉不对`, or similar wording without expected behavior.
- 请求混合排查和实现，例如一边问是否是 bug，一边暗示修法。
- 同一 workspace 里多个兄弟仓库名称相近或存在复制模块。
- 改动触及资金、结算、用户状态、权限、安全中间件、API 契约、任务调度、Redis shared keys、大表查询或迁移。

## 确认方式

- 问一到三个具体问题，不要泛泛说“请澄清”。
- 带上已经核对过的源码事实，让用户能纠正准确假设。
- 可行时给出最小安全实现边界。
