# 需求意图闸门

## 工作流程

1. 先判断最新用户请求类型：只分析、回答问题、允许实现、代码审查、验证或交接；以最新明确指令为准。
2. 先收集源码证据。读取生效的 `AGENTS.md`、仓库内 AI 文档、相关代码路径、文档、路由、配置、测试、日志或用户粘贴的 SQL/payload。
3. 非平凡任务先说明工作理解：目标行为、当前源码行为、可能入口、可能文件、非目标和验证计划。
4. 当需求存在多个合理业务含义、目标仓库/入口不清，或改动会改变 API 契约、数据、权限、安全、大查询、缓存兼容、迁移、补偿、用户可见流程时，编辑前先问一个简洁确认问题。
5. 不问仓库里能低成本查到的问题；先搜索和阅读，只问剩余歧义。
6. 尊重只分析语言。用户说 `先分析`、`先不改代码`、`只给方案` 等时，在用户明确切换到实现前不改文件。
7. 如果进入实现，把已确认边界传递给当前仓库已安装且适配的专项 skill 或交付复核流程。
8. 如果无法可靠理解，停在修改前，报告阻塞点、已检查证据、猜测风险和需要确认的问题。

## 需要确认的信号

满足任一项时，编辑前确认：

- 同一措辞可对应不同业务流程、表、角色、项目或入口。
- 用户只描述现象，没有说明期望行为。
- 粘贴的日志、SQL、payload 或截图可能来自旧运行时，而不是当前源码。
- 多个兄弟仓库存在相似文件，目标仓库不明确。
- 修复可能改变权限、RouteMeta、业务码、签名/加密、Redis Key 兼容、表结构、数据修复、任务调度或前端路由。
- 实现需要破坏性改动、数据迁移、回填、缓存失效或补偿。

## 意图摘要格式

内部使用；风险较高时在用户更新里给短版：

```text
我理解的目标:
源码证据:
仍不确定:
拟修改边界:
不会修改:
确认问题:
```
