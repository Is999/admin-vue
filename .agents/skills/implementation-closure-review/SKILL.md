# 实现闭环复核

## 工作流程

1. 根据最新用户要求、历史约束和修改前检查过的源码证据，还原用户真实需求。
2. 确认最终实现仍匹配已确认意图；如果意图含糊且未确认，不要称为完成，要标记风险。
3. 交付前建立闭环矩阵：需求项、源码入口、修改文件、验证证据、剩余风险。
4. 端到端追真实调用链。生产入口未接上时，不能只凭 helper、DTO、UI 壳或孤立测试声称完成。
5. 检查当前前端仓库真实存在且受影响的交付面：page、route、component、drawer/modal、table/form、store/composable、API wrapper、TypeScript type、i18n、permission、UI 状态、测试和交付说明；不存在的面标记不适用。
6. 实现后复核命名和结构：目录、文件、变量、函数、方法、常量、结构体、接口和字段必须匹配业务职责和当前抽象层级。
7. 拒绝半成品收口：脚手架、样例、单一分支、happy path 或未验证接线不能描述为完成；无法完成时明确未完成项、阻塞、影响和下一步。
8. 最终说明要诚实区分：已完成、已验证、已跳过、被阻塞、需后续处理。

## 命名规则

- 优先使用短而准确的领域名，避免长解释型命名。
- 避免 `handleData`、`doProcess`、`DefaultValue`、`CommonConfig`、`Manager`、`Helper`、`Info` 等泛名，除非本地代码已赋予明确含义。
- 实现职责变化时同步改名，不保留描述旧行为的名字。
- 常量按领域前缀和稳定业务含义分组。
- 结构体用于领域对象；符合仓库风格时，请求用 `Req`、响应用 `Resp`、列表项用 `Item`。
- 只有真实替换点才抽接口，不为了显得抽象而新增接口。
- 目录和 package 名要匹配归属，不创建装杂物的 catch-all 目录。

## 闭环矩阵

非平凡任务内部使用这个形状，并在最终交付中摘要：

```text
需求项:
确认意图/依据:
入口/调用链:
实现文件:
同步面:
验证:
状态: completed | skipped | blocked
说明:
```
