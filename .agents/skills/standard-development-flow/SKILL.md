# 标准开发流程

此 skill 是主流程，负责调度专项 skill，但不替代专项 skill。

## 阶段闸门

1. 入口闸门：先判断最新请求是只分析、实现、审查、验证还是交接；意图或边界不清时使用 `$requirement-intent-gate` 并暂停编辑。
2. 源码闸门：编辑前读取生效的 `AGENTS.md`，并用 `rg --files` 发现当前仓库实际存在的 AI 文档、代码路径、测试、路由、配置、契约和文档。
3. 计划闸门：非平凡任务先说明已确认目标行为、源码证据、可能文件、非目标、风险面和验证命令。
4. 实现闸门：改动保持小而聚焦，优先使用当前仓库现有模式，并按当前仓库 `AGENTS.md` 列出的已安装专项 skill 选择执行。
5. 后端契约闸门：纯前端任务只执行前端规则；任务跨到后端仓库时，到对应后端仓库读取其 AGENTS.md 和 repo-local skills。
6. 同步闸门：只同步当前前端仓库真实存在且受影响的 route、menu、component、API wrapper、TypeScript type、i18n、permission、UI 状态和测试；后端契约变化到后端仓库处理。
7. 领域闸门：按当前仓库真实技术面调用已安装专项 skill。Redis、配置/热加载、worker/队列、安全路由、迁移/回填、前端 CI/hook 或兄弟仓库同步不存在时，不强行套用。
8. 可维护性闸门：持续执行 AI 开发规范：稳定契约和关键逻辑补简洁中文注释，命名短且准确，结构可读，无隐式魔法，无半成品脚手架。高风险或全仓库场景使用当前前端仓库已安装且适配的可维护性检查。
9. 收口闸门：非平凡任务交付前使用 `$implementation-closure-review`；需要 staged 文件或运维交接时使用 `$release-handoff-guard`。复核需求闭环、命名、真实调用链、同步面、测试、跳过项和剩余风险。
10. 交付闸门：最终说明必须区分已完成、已验证、已跳过、被阻塞、需后续处理，并说明迁移、回填、缓存、补偿需求。

## 必留证据

每个实现任务都要留下足够证据，便于其他开发者复审：

- 已检查的源码事实：文件、函数、路由、文档、配置、SQL/log 片段或测试。
- 实现边界：改了什么、明确不改什么。
- 同步面：列出当前前端仓库实际存在且受影响的 route、component、API wrapper、type、i18n、permission、UI 状态和测试；不存在的面写不适用。
- 验证：实际命令和结果；验证失败时使用当前仓库已安装且适配的专项 skill；跳过检查必须说明原因。
- 仓库状态：每个触碰 Git 仓库都运行 `git diff --check` 和 `git status --short`。

## 专项路由

以当前仓库 `AGENTS.md`、`.agents/skills` 和实际文件树为准，只调用真实适配且已安装的专项 skill。缺失或不适配时按同样闸门手动执行并说明。

## 团队落地

把此流程落到其它仓库或团队成员环境时，若存在 `references/team-adoption.md`，先读取它。

大型任务需要结构化交接时，读取 `references/delivery-template.md`。
