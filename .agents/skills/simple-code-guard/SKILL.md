# 简洁代码护栏

在业务正确、生产闭环的前提下，把简洁、逻辑清晰、易读、易维护和高性能放在最高优先级。只解决当前代码和需求已经证明的问题，不为假设场景预埋复杂度。

本 Skill 只补充复杂度判断，不能覆盖项目开发规范：

- 代码必须简洁、逻辑清晰、易读易维护；合理封装只能用于降低真实复杂度、复用稳定边界或隔离外部依赖，不能过度封装破坏清晰性。
- `admin`、`api` 新增或重构的变量、常量、方法、结构体、字段、map 字段、路由、GORM Model、接口和关键逻辑必须补中文注释。
- `admin-vue` 新增或重构的变量、常量、函数、类型、接口、字段、路由、状态结构和关键交互必须补中文注释。
- 简洁要求用于控制注释长度和准确性，不能作为省略必需注释的理由。

## 读取项目边界

1. 先服从系统、开发者和用户指令。对每个范围内仓库读取从工作区到目标目录的全部 `AGENTS.md`，同级规则冲突时以离目标文件最近者为准；AI 开发提示词和 AI 开发规范用于补充项目细节，不能覆盖生效的 `AGENTS.md`。
2. 只读取当前项目对应参考：
   - `admin`：`references/admin.md`
   - `api`：`references/api.md`
   - `admin-vue`：`references/admin-vue.md`
3. 从真实入口追到最终副作用、状态变化和测试，不凭文件名或历史结论判断。

## 简化判断

每个新增函数、接口、结构体、状态、分支和包装层都必须由明确需求、真实调用方、失败证据或稳定复用边界支撑；最终候选中没有需求依据和真实调用方的代码不保留。

优先按以下顺序处理：

1. 删除无调用方代码、未上线兼容逻辑、重复实现和永远不可达分支。判断“无调用方”必须同时检查静态引用、运行时注册或反射入口、对外契约，以及当前部署或运营状态。
2. 内联只转发参数或只调用下一层的无语义包装，缩短主调用链。
3. 用提前返回和直接条件简化嵌套，合并表达同一业务规则的重复判断。
4. 保留具体类型；只有需要隔离外部依赖、解除真实依赖环或已有多个实现时才增加接口。
5. 只在重复逻辑稳定或提取后明显降低主流程阅读成本时增加 helper。
6. 命名直接表达业务动作或状态；职责变化时同步改名，不保留历史错名。

禁止为了可能复用、可能替换或极低概率假设增加层级。不要把 `A -> execute` 改成连续的 wrapper、helper、internal 跳转。

## 正确性边界

- 重构默认不修改现有外部可观察行为，以及返回值、错误、事务、鉴权、幂等、重试和状态机语义；需求明确要求改变时除外。
- 不降低 `AGENTS.md`、AI 开发提示词和 AI 开发规范中的注释、命名、分层、接口同步和验证要求。
- 锁、租约、原子状态迁移、超时、取消、幂等和迁移互斥若对应真实并发或故障窗口，属于业务正确性，不按“理论安全”删除。
- 必要复杂度应贴近使用点，状态数量和跳转层级取满足当前故障模型的最小值，并由聚焦测试证明。
- 项目未上线时，确认无调用方后直接删除旧协议、旧字段和兼容分支，不保留双轨路径。
- 不用大范围重构掩盖局部问题，不顺手改动任务之外的代码。

## 性能判断

- 先确认热路径、数据规模、调用频率和外部 I/O，再决定优化重点。
- 热路径优先减少数据库或 Redis 往返、全量扫描、重复序列化、内存分配、对象创建、接口转换和字符串拼接。
- 能预分配已知容量就预分配；能复用现有解析结果就不重复计算；能批量且有明确上限就不逐条 I/O。
- 冷路径不以难读的微优化换取理论收益。
- 性能优化不得放宽数据边界、错误处理、并发正确性或资源释放。

## 执行流程

1. 写明不变的业务语义、当前真实问题和非目标。
2. 列出入口、主调用链、状态或副作用、现有测试，识别最长跳转和重复点。
3. 先删后改；选择能解决问题的最小差异，不先建框架。
4. 逐项核对当前项目实际存在且受影响的调用方、配置、SQL/Lua、指标、文档和测试，确认没有半成品或失效入口。
5. 对热路径检查 I/O 次数、扫描上限、分配和并发；没有证据时不宣称性能提升。
6. 运行项目参考中的验证，并在最后一次修改后重新检查 diff 和仓库状态。

## 审查输出

每个问题只报告四件事：当前证据、实际影响、最小改法、验证方式。没有证据的问题不落代码；必要复杂度明确说明它保护的真实边界。

完成时必须给出直接结论：业务语义是否保持、生产链路是否闭环、是否仍有阻断项，以及是否需要迁移、回填、缓存失效或补偿任务。
